Servlet 存储在哪儿？

  1. web 容器启动时解析 web.xml 文件，将 <url-parttern> 和对应的完整类名存储在 Map<String,String> 集合中，Key对应

    <url-parttern> 的内容，Value 对应完整类名（以下称为 UrlMap ）。

  2. 实例化的时候会将 <url-parttern> 和 Servlet 实例存储在一个 Map<String,Servlet> 集合中，Key 对应 <url-parttern> 的内容，Value 对应 Servlet 实例（以下称为 ServletMap，由此可见每个 Servlet 在内存中只有一个实例，但这不是单例）。

Servlet 生命周期

不做特殊配置的情况下：

  1. 客户端首次访问 URL，Tomcat 容器截取 URL 后面的内容；
  2. 在 ServletMap 集合中寻找是否有对应的 Servlet 实例，如果有直接调用 Servlet 的 service() 方法；
  3. ServletMap 集合中没找到：
    1. UrlMap集合中寻找 url 对应的完整类名；
    2. 通过反射执行无参构造函数获得Servlet的实例（将 <url-parttern> 和 Servlet 实例存储在 ServletMap 中）；
    3. 执行 init() 方法（初始化 Servlet 的时候执行）；
    4. 执行 service() 方法。
  4. 后续访问只执行 Servlet 的 service() 方法；
  5. 在容器关闭、项目重新部署或者长时间没有访问某个\ Servlet 的时候，会先执行 destory() 方法，然后销毁 Servlet 实例。

特殊配置：在 web.xml 中加上 <load-on-startup> 标签，可以在启动时便构造实例和执行 init() 方法。

多线程问题

Servlet 是伪单例模式（真单例的构造方法必须是私有的），但是运行的时候是在多线程的情况下运行的，多个用户可以同时访问，所以避免使用全局变量，会存在线程安全问题。

GenericServlet

由于每次使用 Servlet 都需要实现 Servlet 接口，则我们必须在自己的 Servlet 中重写 Servlet 的所有方法：init()、service()、destory() 和 getServletInfo()，但是我们大部分时候只需要在 service() 方法里编写业务功能，所以另外三个方法代码重复，因此使用一个中间类 GenericServlet 来实现 Servlet 接口：

  • 其中 service() 方法被声明为 abstract 抽象的，子类就只需要重写这个方法就能实现 Servlet 了；

  • 重载了 init(ServletConfig config) 方法并把 config 给成员变量；同时，调用了一个默认为空的 init() 方法，原因：

    • 它把 config 赋值给成员变量，并通过 getServletConfig() 方法展现给子类，以方便子类使用；
    • 空的 init() 方法可以开放给子类，让子类自由发挥，子类重写 init() 方法即可。

  • 然后我们的 Servlet 可以通过继承 GenericServlet 实现；
